# 🚀 四. 使用 Vercel 部署项目

Vercel 作为 Next.js 的开发商，在免费计划中提供了非常充裕的额度，对于我们目前的博客系统来说，访问量几乎永远不会触及免费额度的上限。因此，利用 Vercel 来白嫖免费额度进行建站，可以说是最省心、最完美的选择！🌟

## 🛠️ 提前准备步骤

- **第一步：** 注册一个 Vercel 账号。
- **第二步：** 将我们的项目推送到 GitHub 仓库。Vercel 对 GitHub 的支持非常完美，可以轻松实现代码的自动构建和 Hooks 触发。
- **第三步：** 在 Vercel 控制台中选择 **Add New Project**，然后直接导入我们的 GitHub 仓库。
  > ⚠️ **注意：** 对于 Next.js 项目，Next.js 的版本必须大于 `15.3.8`（较低版本存在严重的安全漏洞）。
- **第四步：** 选择项目的框架类型（通常会自动识别为 Next.js），点击部署，项目就成功创建啦！整个过程非常顺滑！🎉

## 🗄️ 数据库连接 (Neon + PostgreSQL)

对于初次接触后端或全栈项目的同学来说，刚开始可能会有一点疑惑：**数据库到底应该放在哪里？难道要和前端项目放在同一台服务器上用 Docker 启动吗？** 🤔

其实，在现代的云原生开发中，数据库通常由第三方的 Serverless 数据库服务商提供。通过简单的连接字符串，我们就能轻松连接上数据库。

在这个项目中，我使用的是 **Neon** 🐘，一个非常优秀的开源第三方数据库服务商。Neon 完美支持 `Prisma + PostgreSQL` 的组合。为此，我们需要先将本地的数据库引擎切换为 PostgreSQL。

在 Neon 上创建好数据库后，它会提供一个连接字符串供我们使用，大概长这样：
```bash
postgresql://neondb_owner:xxx...
```

将其复制并配置到 Prisma 的 `datasource` 中即可。
连接好后，在本地终端执行 `pnpm prisma db push` 来检测是否能成功连接上对应的数据库。
> 💡 **小贴士：** 完成这一步后，建议不要在本地直接去操作线上数据库的内容，而是通过云端控制台或 Prisma Studio 进行操作，以防出现意外。

## ⚙️ 修改构建脚本和参数

虽然连接上了数据库，但在 Vercel 部署时会遇到一个问题：每次 Vercel 部署项目时，都会严格执行 `pnpm run build` 命令。然而，当前项目在云端构建时，并没有自动生成对应的 Prisma 客户端。因此，我们需要修改一下 `package.json` 中的打包命令。

```json
{
  "scripts": {
    "dev": "next dev --turbopack",
    "build": "prisma generate && prisma migrate deploy && next build",
    "postinstall": "prisma generate",
    "start": "next start",
    "lint": "next lint"
  }
}
```

**修改说明：**
我们使用了 `&&` 来串行执行命令，确保在每次部署时，按顺序完成以下操作：
1. **`prisma generate`**：在当前环境生成一个 Prisma 客户端，以便程序能够正常操作远程数据库。
2. **`prisma migrate deploy`**：将本地开发阶段通过 `migrate dev` 生成的迁移记录，按顺序推送到云端数据库中，保证数据库结构的同步。
3. **`next build`**：最后再执行 Next.js 的常规构建。

## 🌍 不同环境的变量配置

因为我们同时拥有“云端生产数据库”和“本地开发数据库”，所以我们需要合理规划 `.env` 环境变量文件，确保本地连接测试环境，生产连接生产环境。Next.js 的环境变量加载优先级如下：

| 优先级 | 文件名 | 说明 |
| :---: | :--- | :--- |
| **1 (最高)** | `.env.$(NODE_ENV).local` | 如 `.env.development.local`、`.env.production.local` |
| **2** | `.env.local` | 所有环境通用的本地覆盖文件（通常包含敏感信息，不提交 Git） |
| **3** | `.env.$(NODE_ENV)` | 如 `.env.development`、`.env.production` |
| **4 (最低)** | `.env` | 所有环境的默认值兜底 |

> `NODE_ENV` 指的是我们运行代码时的环境模式（development / production 等）。

在 Vercel 平台上，我们可以直接通过 **Environment Variables** 设置来添加全局环境变量。为了信息安全，**千万不要把包含密码等敏感信息的 `.env` 或 `.env.local` 文件提交到 Git 仓库上！** 🙅‍♂️ 所有的生产机密信息都应该在 Vercel 控制台里手动配置。

## 🔐 针对 Better-Auth 的生产环境修改

将 Better-Auth 部署到 Next.js 生产环境时，配置会稍微复杂一点点。

**1. 数据库适配器：**
刚开始部署时，由于我把 Better-Auth 的引擎改为了其他引擎，导致应用一直报 `500` 错误 💥。查阅文档后发现，在 Next.js + Prisma 中使用 Better-Auth，必须引入专用的适配器才能流畅运行：
```typescript
import { prismaAdapter } from "better-auth/adapters/prisma";
```

**2. 跨域和 BaseURL 设置：**
我们需要将项目的 `baseURL` 添加到 Better-Auth 的 `trustedOrigins` 防跨域设置中。

**完整配置参考：**

```typescript
import { betterAuth } from "better-auth";
import { prismaAdapter } from "better-auth/adapters/prisma";
import { admin } from "better-auth/plugins";
import { prisma } from "@/lib/prisma";

const siteOwnerEmail = process.env.SITE_OWNER;

export const auth = betterAuth({
    baseURL: process.env.BETTER_AUTH_URL,
    trustedOrigins: [
        process.env.BETTER_AUTH_URL!,
        process.env.NEXT_PUBLIC_BETTER_AUTH_URL!,
    ].filter(Boolean),
    database: prismaAdapter(prisma, {
        provider: "postgresql",
    }),
    socialProviders: {
        github: {
            clientId: process.env.GITHUB_CLIENT_ID as string,
            clientSecret: process.env.GITHUB_CLIENT_SECRET as string,
        },
    },
    plugins: [admin()],
    databaseHooks: {
        user: {
            create: {
                before: async (user) => {
                    if (!siteOwnerEmail) {
                        throw new Error("SITE_OWNER is required");
                    }
                    return {
                        data: {
                            ...user,
                            role: user.email === siteOwnerEmail ? "admin" : "user",
                        },
                    };
                },
            },
        },
    },
    //...
});
```

> ⚠️ **巨坑警告：** 
> Vercel 每次自动部署生成的带 hash 值的临时预览域名，是**无法**直接用来充当 Better-Auth 的 `baseURL` 的！**必须**使用 Vercel 初始分配的固定项目域名（或你绑定的自定义域名）作为环境变量中的 `BETTER_AUTH_URL`，否则会出现认证失败的问题。
