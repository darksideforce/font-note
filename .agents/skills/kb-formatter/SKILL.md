# AI 文件格式化与润色规范

> **【系统指令】**
> 当用户请求对文件进行格式化、排版或错别字纠错时，AI 必须严格按照以下流程执行文件覆写。

## 执行流程

1. **定位并读取目标文件**：
   - 优先使用用户在对话中明确指定的文件名或路径（如“排版一下 React笔记”）；
   - 若用户仅提供了模糊的主题关键词，使用搜索工具在知识库中找到对应文件；
   - 若用户未提供任何指向性信息，则默认读取用户当前正在打开（Active Document）的文件。
2. **术语与错别字审查**：
   - 严格审查和纠正所有不规范的技术名词（例如：游览器 -> 浏览器，弱缓存 -> 协商缓存，响亮 -> 向量）。
   - 修正日常错别字和语病，使行文符合专业开发者的严谨度。
3. **Obsidian 属性与特殊语法保护**：
   - **识别 YAML Frontmatter / Properties**：如果文件顶部存在 `---` 包裹的属性区，必须将其视为 Obsidian Properties，不得当作正文标题、分隔线或普通 Markdown 段落处理。
   - **保留并规范属性区**：允许整理缩进、列表格式和明显错别字，但不得删除已有属性、改名字段或改变字段含义。常见字段包括 `aliases`、`tags`、`status`、`created`、`updated`、`cssclasses`。
   - **缺失属性时补齐最小模板**：如果目标文件是知识库笔记且完全没有属性区，可在文件第一行补充最小属性模板：
     ```yaml
     ---
     aliases: []
     tags: []
     status: 待完善
     created: {{date}}
     ---
     ```
     其中 `{{date}}` 应替换为当前日期，格式为 `YYYY-MM-DD`。
   - **属性语法必须合法 YAML**：数组可以使用块列表，示例：
     ```yaml
     tags:
       - JavaScript
       - Obsidian
     aliases:
       - 别名
     ```
     不要在 YAML 的 `tags` 字段里写成 `#tag`，应写成 `tag` 或列表项；正文内联标签才使用 `#tag`。
   - **保护 Obsidian 特殊语法**：不得破坏 `[[内部链接]]`、`[[笔记名|显示文本]]`、`![[嵌入内容]]`、`> [!note]` callout、`#tag` 标签、`%%注释%%`、`==高亮==`、`$LaTeX$`、Mermaid 代码块、块引用 ID（如 `^block-id`）。
   - **不要误格式化特殊语法**：不要给 Obsidian 语法额外套反引号，不要把 `[[...]]` 改成普通 Markdown 链接，不要把 callout 的 `> [!type]` 拆散。
4. **Markdown 规范化排版**：
   - **盘古之白**：在中英文之间、中文与数字之间强制增加空格。
   - 修复混乱的缩进、错误的列表格式（如 `-` 后面缺少空格）。
   - 确保所有代码块都带有正确的语言标识（如 ` ```typescript `）。
   - 梳理标题层级 (`#`, `##`, `###`)，使内容层次分明。
   - **克制地添加 Emoji**：在标题、列表项或重点段落处，克制地加入一些契合语境且不刺眼的 Emoji（如 💡、📌、应对复杂性的 🧩 等）以增强视觉层次感，切忌过度使用。
   - **尊重原有结构**：严禁擅自修改原文的段落结构、合并或拆分长列表、增删原有的标题层级。只在原有的结构框架内进行润色、纠错和排版。除非用户明确要求，否则绝对不要改变原有的行文布局。
   - **保留特殊语法**：绝对不能修改、格式化或破坏 Obsidian 的特殊语法（例如以空格加井号开头的标签 ` #tag`）。不要为它们添加代码块反引号（` ` `），保持原样忽略即可。
5. **阅读体验增强（仅在用户明确要求时启用）**：
   - 当用户提出“提升阅读体验”、“优化布局”、“重点更突出”、“更好读”、“美化排版”等诉求时，必须主动通读正文，识别 3-7 处真正值得凸显的关键内容，而不是只做机械空格和错别字修正。
   - 可优先标识这些内容：核心结论、定义、易混淆边界、步骤中的关键动作、风险/限制、因果关系、对比结论、作者反复强调的判断。
   - 使用 Obsidian 兼容的轻量强调手段：`**加粗**` 表示关键术语或核心判断；`==高亮==` 表示最需要回看的结论；短表格表示清晰对比；分隔线或小标题只用于原文已有明显段落边界的地方。
   - **默认不要新增 Obsidian callout**：不要主动写 `> [!tip]`、`> [!warning]`、`> [!summary]` 这类提示/总结块，尤其不要为了“阅读体验”凭空加记忆方式、核心结论、最终理解等 callout。只有用户明确要求使用 callout，或原文已经存在同类 callout 且需要整理时，才允许保留或微调。
   - 若当前笔记已经使用 `cssclasses` 或 HTML 样式片段，可沿用既有风格；否则不要凭空大量加入 HTML、内联颜色或复杂 CSS。确需用颜色时，只允许少量使用 Obsidian 可渲染且语义清楚的 `<span style="color: ...">...</span>`，并避免把整段文字染色。
   - 不要把每个关键词都加粗，不要连续堆叠 Emoji、加粗、高亮和颜色。每个段落最多选择一种主要强调方式，保证重点真的稀缺。
   - 允许在不改变原意的前提下，把过长段落拆成更易读的短段，或把密集枚举整理为列表/表格；但不得删减信息、改变论证顺序，或重写成与原文风格明显不一致的新文章。
6. **术语笔记待办**：
   - 不要在正文中顺手把普通专业术语改成 `[[ ]]` 双链，避免制造大量没有实际沉淀价值的孤儿链接。
   - 如果发现值得后续沉淀的核心专业术语，应在笔记底部新增或更新 `## 待创建术语笔记` 小节，用任务列表记录需要补建的术语笔记，并且使用 `#待创建术语笔记` 标签来进行标识。
   - 待办格式示例：
     ```markdown
     ## 待创建术语笔记

     - [ ] [[有限状态机 (FSM)]]：解释它在本文流程中的作用。
     - [ ] [[乐观更新 (Optimistic Update)]]：补充前端状态更新与失败回滚机制。
     ```
   - 只有当正文中已经存在稳定的目标笔记，或用户明确要求建立双链时，才在正文中使用 `[[ ]]`。
7. **静默覆写**：使用写文件工具将排版好的内容直接覆盖原文件。
8. **总结汇报**：完成覆写后，必须向用户清晰地展示修改结果清单，明确指出修改了哪些地方及具体内容。例如：“列出纠正的错别字或技术名词的具体清单”、“列出新增或整理了哪些 Properties 字段”、“列出底部新增了哪些待创建术语笔记”、“列出对哪些段落或标题添加了 Emoji、加粗、高亮、表格或颜色等视觉修改”。确保用户对文件的变动一目了然。
