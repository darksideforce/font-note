# 引擎做了一些什么？
在一些比较复杂的harness中，为了架构的方便，需要将决策层和执行层分离，决策层指的就是大脑，负责的是根据上下文以及输入来决定下一步应该做什么，而执行层则是负责的具体去做这个行为。

几个字段需要注意的是：
event：给前端通信的事件，比如通知前端需要输入字段
type：用来决定执行什么执行器的
state：内部记忆，用来交给大脑用的，相当于只读快照
执行器：执行通过大脑解析的指令的具体函数

## 基础实现步骤：
- 初始化引擎，接受外部传入的大脑和配置信息
- 使用策略模式初始化执行器，约定若干个字段对应`call_llm``call_tool``request_human_approve``request_human_prompt``finish`等
 **step方法**
- 接受一个state，用于和大脑之间互相通信，并记录通信信息
- state记录通信次数
- 检查通信次数是否超出上限，超出上限就中止通信，防止死循环（死循环熔断机制）
- 当调用者通知引擎这是一次用户同意工具调用的通信时，伪装成大脑的回答（为了直接去调用执行器）
- 当调用者通知引擎这是其他调用方式时，通知大脑进行解析出具体指令（类似call_llm或者tool_call这种）
- 开始把解析出来的指令使用对应的执行器来执行
- 批量的工具审批需要挨个执行
- 将执行器的结果，解析出event、context等信息，推入信息返回给调用者
- 如果执行器的结果是等待用户审批，则中断循环而等待下一步指示
- 结束这次执行，等待下次执行
