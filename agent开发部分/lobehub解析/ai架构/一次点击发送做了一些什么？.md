# 💡 简单介绍

- 点击页面的发送。
- 调用位于 `ConversationStore` 的 `sendMessage`。
- 在 `Store` 内进行预处理，执行传入的 `onBeforeSendMessage` hooks，并且取出 `message`。
- 做好预处理后，调用位于 `ChatStore` 的 `conversationLifecycle` 内的 `sendMessage`。`ChatStore` 是全局单例，并且以一种无头引擎的方式来给调用方提供服务。
- `sendMessage` 内进行以下操作：解析指令、预加载技能和工具、乐观更新数据、队列处理数据。
- `sendMessage` 做好前端部分后，调用 [[agent开发部分/lobehub解析/ai架构/agent runtime - 大脑|Agent Runtime]] 的接口来进行发送。

# ⚙️ 状态机 (State Machine)

第一阶段我统称为 **实体创建期**：一次性落盘用户消息实体和 AI 占位实体，确保前端 UI 的气泡坑位和后端数据库的主键严格绑定。

第二阶段我称为 **流式填充期**：将大模型的 [[长链接 (Long Polling/SSE)]] 响应与先前的 AI 占位实体绑定，单向注入流式内容。

> 这样设计最大的好处是：大模型网络最不稳定、最容易超时的阶段，被严格隔离在第二状态机中。就算大模型完全崩溃，用户的数据（第一状态机）也绝对安全，且随时可以精准复用原有的占位气泡进行 Retry。

- 发送状态机
- 流式填充期状态机

# 📝 conversationLifecycle 文件解析

现在关心 `sendMessage` 的前端部分处理：

- 取出传入的 `editor` 对象，并先取出选择的 `skills`、工具和选择的 `agents`。
- 取出本地文件的引用。
- 判断是否是新进程（子对话）。
- 判断当前是否有指令，如果有指令，例如压缩或者新开对话，则中断 `sendMessage` 并执行特定方法。
- 如果有文件，则按本地文件迅速上传到后端并返回文件 ID。 #待补充如何实现
- 如果有选择技能和工具，则按技能和工具的 ID 真实地将解析的技能和工具内容进行注入。 #待补充如何实现
- 判断当前有无信息正在输出，有的话则将当前用户的输入塞入消息队列 (`queuedMessages`) 并立刻中断发送流程。
- 抓取当前对话的历史聊天记录，并使用滑动窗口、上下文压缩、RAG 向量提取的方式来精准提取上下文。 #待补充如何实现
- 取出新进程的父对话 ID。
- 使用乐观更新，立刻生成一个假 ID 并进行占位，将状态机流转到运行中，并分配唯一的状态 ID（这里的状态 ID 表示一次用户和 AI 的对话，不是一整个对话）。
- 更新页面，立即在对话框中显示消息，创建临时消息，创建一个 AI 的 loading 态。 #待补充如何实现
- 将消息链接到两个索引：单个 `messageOperationMap` 表，方便追踪消息对应的是哪一次对话（状态机）；`operationsByMessage` 表，代表一次 `message` 可能会有多次状态（比如派生的任务）。
- 将用户输入的信息进行一次备份，提取原始的 JSON 进行备份。
- 准备进入发送给后端阶段。（gateway模式到此为止）
- 发送后端阶段：**插件说明书去重**。发消息前先翻旧账。如果当前使用的插件在历史聊天里已经给大模型发过一次说明书了，就不再发第二遍。只把第一次出现的新插件的说明书拼在消息后面。
- 发送后端阶段：使用后端返回的 `topicId` 来替换掉之前的假 ID，并更新左侧对话列表的状态，使用后端返回的全量消息数组来替换当前的消息数组。
- 发送后端阶段：将侧边对话列表的高亮转为当前的对话。
- 发送后端阶段：如果此时失败，则将状态机流转到失败，并回复用户的消息。
- 清空草稿态，因为发送后端状态已经完成。
- 判断是否是新对话（或无标题对话），是的话调用 AI 来进行总结标题（测试态可以忽略该行为）。 #待补充如何实现
- 流转该发送状态机至已结束。
- 提取所有属于当前对话的信息，关闭当前所有等待用户授权或等待状态的工具。
- 将上传的文件转换为文件 ID 等前置上下文，由后端进行拦截并发送给 AI 完整的文件上下文。 #待补充如何实现
- 将前置上下文和其他信息进行组装，并完整发送给 AI 侧，此次流程结束。

# 🧠 准备调用 AI 侧部分逻辑 #gateway
topicid？context？任务队列是如何被监听到的，如何被管理的？

- 提取出websocket链接，准备后续用于和后端通信
- 判断是否为新建的对话
- 提取出taskId
- 如果是新建的对话，防御性提取出对话需要使用的message等context
- 提取出用户点击中断的标识
- 发起请求，通知后端创建好对应的执行任务，并将上下文发送给后端，接受后端返回的任务id以及对话id、占位信息id等
- 判断用户有没有在请求期间点击中断，如果点击了则通知后端销毁任务 #待补充如何实现 
- 如果该次对话为新建对话，通知store销毁临时的变量，请求数据库拉取最新的message并更新到ui上，最后再切换对话到新的对话
- 强行对齐上下文，通知ui显示loading
- 创建前端侧的状态机，并流转状态至拉起服务端智能体
- 将状态机与服务端返回的占位信息id进行绑定到字典，方便后面进行查询（一次状态机对应一次对话，但是一次对话可能对应多个状态机）。
- 组装好事件分发器，该事件分发器将后端返回的结果接入不同的处理逻辑中 #待补充如何实现 
- 通知后端准备建立sse连接，发送刚才后端返回的任务id，token等，并将上下文传入，并注册事件监听，中止事件回调等 #待补充如何实现 

# 🧠 准备调用 AI 侧部分逻辑 #client

这一步是所谓的网关层，即用户自己决定远程的api，并在前端完成状态机等逻辑

- 判断该流程是否已经存在 `operationId`，如果有则直接进行重新请求。
- 如果没有则创建新的 `operationId`，准备启动新的状态机，该状态机针对的是 AI 流程。
- 从 `Store` 中提取出运行上下文、提到的工具、状态对象、大模型配置单。
- 提取出当前选择模型的上下文限制。
- 将刚才准备好的数据创建出 [[agent开发部分/lobehub解析/ai架构/agent runtime - 大脑#大脑|大脑]]（Agent 持续运转的决策核心部分）。 #待补充如何实现
- 创建出引擎，注入上一步创建的大脑、执行工具、状态机查询器、中断机制 (`AbortController`)。 #待补充如何实现
- 进入 Agent 核心循环 (`while`) 逻辑：
  - 顶部中断检查：判断是否用户点击中断，如果触发了中断则跳出。
  - 战前情报收集：从历史记录中提取最新的 `todo`（任务计划），强迫模型按照 `todo` 来做。
  - 将 `todo` 等动态情报并入 `context`，并发送给引擎的 `runtime.step()`，引擎会去通知大脑执行下一步。
  - 这是一个阻塞等待过程，等待这一轮底层网络通信流式输出彻底完成。
  - 跑完一步后处理副作用 (Effects)：如果此时出现报错就在 UI 显示红框，如果需要高危操作授权就触发系统桌面弹窗等。 #待补充如何实现
  - 如果出现没有返回值（上下文断层），则强行中断防死循环。
  - 底部兜底检查：刚刚从漫长网络请求中苏醒，再次判断在等待期间用户是否点击了中断，如果是则清理并跳出。
  - 将新的上下文覆盖旧的 `context` 和 `state`。
  - 此轮 `while` 循环结束。
- Agent [[生命周期 (Lifecycle)]] 结束，开始打扫，检查有没有后置的钩子 (`afterCompletionCallbacks`) 并按顺序进行触发。
- 检查积压队列：检查刚才 AI 循环时用户有无发新消息，如果有则合并消息并延迟触发一次新的发送请求。
- 流转状态机到已完成 (`done`) 或报错 (`error`)，并更新 UI。
- 进行桌面通知。
