# 为什么需要上下文中转
架构背景：在已经拥有websocket客户端的前提下，我们需要一个统一的事件处理器，它不会关心数据是从哪个地方传来，它只关心如何将数据以约定的方式传给前端，通知前端需要如何进行渲染。这样可以比较方便的接入sse和websocket甚至更多的交互协议

# 链路：
- 取出后端的状态机id，准备储存。
- 取出后端返回的对话id，准备用于更新对应的ui区块
- 设置变量，对应思考链条的具体数据和对应id，ui接收到以后会专门显示对应的思考中区块。
- 接受后端传来的event，使用并发调度器进行调节。
- 接收到**stream_start**事件，后端通知开始进行推流
- 上下文中转站进行清空旧推理数组，渲染新对话气泡，绑定新气泡id至对话id
- 接收到**stream_chunk**事件，检查event发现是推理事件，将推理数组显示出来并渲染思考过程
- 接收到**stream_end**事件，本轮推理结束，大模型推送挂起休息
- 接收到**tool_start**事件，后端通知前端准备调用本地工具
- 前端将执行结果发送给后端
- 接收到**tool_end**事件。后端通知前端结束气泡
- 接收到**stream_start**事件，重新开始下个循环
- 接收到**agent_runtime_end**事件，后端通知前端该任务完成。


# 不同分支的处理：
## stream_start
这个分支的意思代表后端已经开始了新一轮的**Assistant**消息记录。
所以作为事件处理器，我们必须做以下这些事：
- 清空各种状态，让这个气泡以一个全新的姿态进行展示
- 通知前端ui开启一个新的气泡
- 将这个气泡与正在输出的对话绑定在一起
- 询问后端，返回后端存储的所有当前这次operation内的所有气泡数据
- 准备将接下来的chunk全部输出在指定气泡内

其他步骤都很简单。重点是询问后端这件事：
为什么要这样做？答案是为了对齐状态，可能会存在有情况导致前几次的气泡没成功渲染出来，我们需要请求后端来尽量保证每一次的ui渲染都是完整的。
你可能会想，那这样岂不是很恐怖，每次都全量更新ui。得意于现代化的虚拟dom，react通过key来进行渲染，当key没变的时候是不会重新进行渲染。而react可以精确的控制渲染的每个步骤，当数组被全量更改完成后，我们再通知react进行ui更新，而react发现其中一些已经存在的气泡并没有更改，所以react只会更新视图中不一样的部分。
 
## stream_chunk
从这进入正常的stream推流，stream普通的推流分为以下三种：
- text：正常的自然语言输出文本
- reason：深度推理文本
- tools_calling：工具调用
- 
这些chunk被设计都通过一个函数进行接受，函数以不同的格式接受以上的数据，最后再函数内进行解析。以统一的方式直接通知zustand的store层，最后再store内去进行ui的渲染。而工具的通知则发送给store进行一个动画的唤起

## tool_start
检查有没有前置钩子，有则进行执行
## toole_end
向后端请求最新的执行结果，并把结果拉取到本地（避免因为websocket出现的各种网络问题而没有同步网络状态）
## tool_execute
工具分为服务端执行和本地执行的两种，该类似的事件就是执行本地的事件

## stream_end
直接清空深度推理的临时数组
## agent_runtime_end
通知总线该次沟通已经完成
强制关闭所有正在进行的动画
强制关闭所有的推理过程
流转状态机至关闭
再次请求后端，要求状态对齐
