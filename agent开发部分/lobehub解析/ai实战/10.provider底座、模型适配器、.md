# Provider 底座、模型适配器 🧩

## 概述

参考文件：`packages/model-runtime/src/core/ModelRuntime.ts`

我们需要先明确几个前提：

- 目前市面上大部分模型供应商都会尽量兼容 OpenAI SDK 或 OpenAI 风格的接口。
- 但“兼容”不代表完全一致：不同 Provider 的 `baseURL`、参数名、流式返回、工具调用、联网搜索、thinking 参数都会有差异。
- 所以 LobeHub 需要一层统一运行时，把外部调用稳定住，再用模型适配器处理不同厂商的方言。

==Provider 底座负责统一外部调用形态，模型适配器负责消化不同厂商的差异。== 这两层组合起来，外部消费者就可以像调用同一个客户端一样调用不同模型。

## 如何提供一个大模型客户端 💡

对于兼容 OpenAI 接口的 Provider，大模型客户端底层通常可以复用 OpenAI SDK。也有一些 Provider 使用自己的 SDK。这里先聚焦 OpenAI-compatible 这条路径，只看为了兼容不同模型和 Provider，需要做哪些事情：

- **拍平请求差异**：不同模型和厂商之间的参数差异，需要在内部先翻译掉，消费者只按统一规范传入数据。
- **拍平返回差异**：大模型返回数据后，也需要统一成一样的格式交给外部。

| 层级          | 主要职责                                       |                                              |
| ----------- | ------------------------------------------ | -------------------------------------------- |
| Provider 底座 | 提供统一运行时、统一 `chat` 等外部 API、统一流式响应包装         | <span style="color: #2563eb;">稳定的调用入口</span> |
| 模型适配器       | 处理不同 Provider 的 `baseURL`、请求参数、流式 Chunk 转换 | <span style="color: #c2410c;">厂商方言翻译器</span> |

## Provider 底座

我们需要实现一个统一运行时。它大概做这几件事：

- 对外提供 API，例如 `chat` 或其他方法，外部不需要关注内部实现。
- 在提交前转换好参数，让整个 `payload` 可以被大模型 SDK 完整消费。
- 把处理后的数据转换成 `ReadableStream` 对象。
- 返回到最外层消费者后，消费者调用 `read` 逐帧读取。
- 提供定制型接口，优先执行外部传入的定制方法，用来覆盖本地默认流处理。

**Provider 底座的重点**：外部只知道“我要调用 `chat`”，至于后面接的是 Qwen、DeepSeek 还是别的兼容端点，由底座和适配器一起处理。

## 模型适配器

有了统一运行时之后，我们还需要把不同模型“压进”这个运行时，这就是模型适配器的职责。

它通常需要处理：

- **Provider 网关地址**：每个供应商的 `baseURL` 都不一致。
- **定制化方法**：一般包括发送前的信息翻译，以及发送后对上游 chunk 的翻译、流转换和 callback 触发。真正发布 Agent `stream_chunk` 的仍然是上层执行器。

基本上，这两套组合起来就可以应对大部分 Provider 接入问题：

```text
外部统一调用
  -> Provider 底座：稳定 chat / stream / Response 形态
  -> 模型适配器：翻译 Qwen / DeepSeek / OpenAI-compatible 方言
  -> OpenAI SDK：真正发送请求并读取上游流
  -> 统一 SSE ReadableStream：返回给外部消费者
```

## 简略代码

```typescript
export const qwenAi = generateCustomRuntime({
    baseURL: '',
    apiKey: '',
    chatCompletion: {
        handlePayload: (payload) => {
            const {
                model,
                ...reset
            } = payload
            return {
                frequency_penalty: undefined,
                model,
                presence_penalty: undefined,
                stream: true,
                temperature: undefined,
                ...reset
            }
        },
        handleStream: qwenStream
    }
})
```
这是一个极简版本，目前只为单次发送做准备。`handlePayload` 会将通用数据格式转换成 Qwen 可以接受的格式，而 `handleStream` 会将上游返回的 Chunk 转换并包装成方便处理的 Stream 流。这样，一个基本的模型适配器就完成了。

- 如果需要针对不同模型修改参数和返回值，基本只需在这一层操作即可。
- Provider 文件可以提供默认的 `baseURL` 等配置；真实用户的 `apiKey` 和自定义 `baseURL` 通常在创建 Runtime 实例时从数据库或客户端配置中传入。这样既方便扩展，也能与底座实现逻辑解耦。




接下来看底座，这次只演示 `chat` 方法：

```typescript
// packages/model-runtime/src/core/openaiCompatibleFactory/index.ts（极简伪代码）

export const createOpenAICompatibleRuntime = ({
  provider,
  baseURL: DEFAULT_BASE_URL,
  apiKey: DEFAULT_API_KEY,
  chatCompletion, // Provider 插件注入：handlePayload / handleStream 等
}: OpenAICompatibleFactoryOptions) => {

  return class LobeOpenAICompatibleAI implements LobeRuntimeAI {
    client!: OpenAI;
    baseURL!: string;
    protected _options: ConstructorOptions;
    private id: string;

    constructor(options: ClientOptions = {}) {
      const apiKey = options.apiKey?.trim() || DEFAULT_API_KEY;
      const baseURL = options.baseURL?.trim() || DEFAULT_BASE_URL;

      this._options = { ...options, apiKey, baseURL };
      this.client = new OpenAI({ apiKey, baseURL }); // OpenAI SDK，指向 Qwen 等兼容端点
      this.baseURL = baseURL;
      this.id = options.id || provider;
    }

    async chat(payload: ChatStreamPayload, options?: ChatMethodOptions): Promise<Response> {
      const inputStartAt = Date.now();

      // ─────────────────────────────────────────
      // Step 1: Provider 定制转换（handlePayload），在这里调用外部传入的 handlePayload 进行转换
      // ─────────────────────────────────────────
      const handledPayload = chatCompletion?.handlePayload
        ? chatCompletion.handlePayload(payload, this._options)
        : { ...payload, stream: true };

      const postPayload = handledPayload;

      // ─────────────────────────────────────────
      // Step 2: 通用 messages 转换（convertOpenAIMessages）
      // 清洗 messages：过滤内部字段、按需转换图片，并按模型处理 reasoning 历史
      // ─────────────────────────────────────────
      const messages = await convertOpenAIMessages(postPayload.messages, {
        model: postPayload.model,
      });

      const finalPayload = {
        ...postPayload,
        messages, // 用清洗后的 messages 覆盖
        stream_options: { include_usage: true },
      };

      // 流式处理配置（callbacks 给 Step 6 用）
      const streamOptions = {
        callbacks: options?.callback, // onText / onFinal / onUsage ...
        payload: { model: payload.model, provider: this.id },
      };

      // ─────────────────────────────────────────
      // Step 3: 真正发送请求（OpenAI SDK → Qwen API）
      // SDK 内部：HTTP POST + 读上游 SSE → 返回 Stream<Chunk>
      // ─────────────────────────────────────────
      const sdkStream = await this.client.chat.completions.create(finalPayload, {
        signal: options?.signal,
      });

      const [prod] = sdkStream.tee(); // 主流给正式响应（debug 分支省略）

      // ─────────────────────────────────────────
      // Step 4: Provider 流转换（handleStream，如 QwenAIStream）
      // SDK Chunk 流 → LobeHub 统一 SSE ReadableStream
      //   convertIterableToStream
      //   → transformQwenStream
      //   → createSSEProtocolTransformer
      //   → createCallbacksTransformer(callbacks)  ← callback 挂在这里
      // ─────────────────────────────────────────
      const sseReadableStream = chatCompletion?.handleStream
        ? chatCompletion.handleStream(prod, {
            callbacks: streamOptions.callbacks,
            inputStartAt,
            payload: streamOptions.payload,
          })
        : OpenAIStream(prod, { ...streamOptions, inputStartAt });

      // ─────────────────────────────────────────
      // Step 5: 包装成 Response 返回给调用方（ModelRuntime / call_llm）
      // body = SSE 字节流，Content-Type: text/event-stream
      // ─────────────────────────────────────────
      return StreamingResponse(sseReadableStream, {
        headers: options?.headers,
      });
    }
  };
};
```

**容易混淆的边界**：`handlePayload` 主要处理 **Provider 参数方言**，`convertOpenAIMessages` 主要处理 **通用 Message 清洗**。这两个阶段不是同一件事。最小纯文本请求可以先简化清洗，但包含图片、Reasoning 历史或工具消息时不能直接省略。

理论上可以直接返回 `ReadableStream`，但当前 LobeHub 的 `chat` 对外契约是 `Response`，所以这里会把 `ReadableStream` 包裹进 `Response` 后再返回。

这里必须确保底座会将 Stream 流正确交给外部传入的 `handleStream` 函数。该函数会组装出一个新的 Stream 流，使其中的回调事件被正确触发。具体可参考下一节。

| 变化 | 说明 |
| --- | --- |

**最终理解**：提交前一共有两轮翻译。第一轮是 Provider 适配器把参数改成厂商能吃的形态；第二轮是底座里的通用转换器清洗 `messages`，再补上运行时需要的通用字段。
