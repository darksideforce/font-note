# 引擎需要的参数

引擎、大脑和执行器之间的作用和具体差别，我们之前已经讨论过了。现在先来说一下我们之前一直忽略的一些东西。

如何在引擎、大脑、执行器之间传递数据？每次 `step` 又如何确定哪些数据需要传递？

我们需要保证有两组数据进入引擎，分别是**运行时数据**和 **`step` 数据**：

- 运行时数据负责控制这一整次 `operation` 对应的信息，一般会用专门的 DB 缓存进行缓存，这里我们叫它 `state`。
- `step` 数据只代表经过上次执行器处理完成后的数据，将立马交给大脑进行分析，这里我们叫它 `context`。

==这些参数是引擎编排里很重要的一环。== 我们来简单看一下这些参数的流转：

```text
AgentRuntimeService.createOperation()      ← 组装初始 AgentState（消息、tools、配置等）
        ↓
executeStep() 从 DB / 内存 load state      ← 每步读取
        ↓
Runtime.step(state, context)               ← stepCount++、forceFinish 等少量改写
        ↓
agent.runner(context, state)               ← Brain：只读，返回 instruction
        ↓
executor 执行指令                           ← Engine：改写 state，产出 nextContext
        ↓
下一步 executeStep(context = nextContext)  ← 外层循环传递
```

遵循这样的路线，每次进入外层的引擎编排层后，编排器会从本地内存或者 Redis 中取出 `state`，而 `context` 则由调用者提供（接口或者执行器）。

在每次执行完成后，`context` 会被再次组装，并通过参数的方式继续传递下去；而 `state` 则会被加工一次后，再次存进 Redis 或本地内存。

初次开始执行的时候，`context` 不由外部传入，而是由调用函数生成。

初次开始执行的时候，`state` 由外部的 `createOperation`（编排层的前置操作）进行组装。这样就完全做到了**组装、执行、推理三个部分各自分离**。

## State

`state` 是持久化运行所需要的基本属性。整个大模型流程的运转中需要众多背景信息提供帮助，否则大模型将寸步难行。

仔细想想，大模型的运转需要什么基本参数？

- `maxSteps`：统计执行了多少步。
- `operationId` 等固定参数。
- `messages`：存储所有信息的数组。
- `status`：标识当前这次 `operation` 运转到了什么状态。
- `usage`：标识这次执行耗费的总 token。
- `tools`：注册的工具。
- `metadata`：存储业务包数据，如 `topicId`、`userId`、`agentId`、`stream` 等基本不会变的数据。这由不同的调用者决定，和业务强相关。

## Context

`context` 中主要是上一次 `step` 执行完毕后生成的 `payload` 或数据，比较简单。每次执行器执行后都会改变 `context`。

它主要用于指导下一次 `step` 做什么，所以它的参数主要和执行器有关：

- `initialContext`：暂时用不到，指的是编辑器等上下文。
- `stepContext`：辅助上下文，包含已注册的 `skills`、工具等。
- `operationId`
- `payload`：代表交给大脑处理的数据。
- `phase`：代表上次执行完成后收到的大模型回复状态码，将交给大脑做决策使用。

> [!summary] 小结
> 我们需要至少确保以上几个东西被传递进引擎和执行器。当然，以上都是最基本的内容，后续还会继续扩展。

# 链路

说完了必须具备的几个参数，我们来说一下现在基础的链路。之前我们已经通过前端发送事件通知了后端，后端在路由中已经接收到了数据，并启动了 `createOperation`。

现在我们来列一下最基本的架构：

```text
第一步：在 AiAgent 入口文件里创建临时的 messageId 和 operationId，并落库 messageId 等。
        ↓
第二步：在 AgentRuntimeService.createOperation() 里落库 operation，并往队列里塞入任务，
        通知系统准备执行一次引擎，同时组装好初次的 context 和 state。
        ↓
第三步：队列监测到推入任务后，立刻新建一个 AgentRuntimeService 类，
        并执行 executeAgent 方法，塞入本地执行器、大脑、引擎。
        ↓
第四步：进入大脑，大脑检查 context，推断出指令 instructions，
        并和 state、context 一起交给引擎。
        ↓
第五步：引擎执行推出的指令，根据指令找到对应的执行器，
        将 context、state 和 payload 交给执行器。
```
