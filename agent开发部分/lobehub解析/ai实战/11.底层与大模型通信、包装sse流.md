# 概述 💡

由于需要适配多个 Provider 以及模型，多个模型之间会有些许差异，而且有的 Provider 需要的字段不同，需要进行翻译；有的 Provider 拥有自己的 SDK。我们需要一层来拍平这些差异。

而且因为 OpenAI 的 SDK 返回的是可迭代对象，这并不是一个方便操作的对象，我们需要进行一次封装。

除此之外，我们还需要提供出一个通用的对象，暴露出类似 `chat`、`createImage` 等对象，方便进行操作。

所以我们需要至少三个函数：

- 拍平差异的 Provider 特有方法：它会针对当前使用的 Provider 来进行翻译。
- 针对 Stream 流的处理函数：将大模型的返回值打磨成规范输出。
- 通用的统一类（适配器）：暴露出通用方法，并且提供特殊的初始化方法来兼容各种处理函数。

# 链条 🔗

# `call_llm` 链路

- `createRuntimeExecutors(ctx)` 创建 executor 映射，`call_llm` 是其中负责模型调用的 executor。
- `call_llm` 接收 payload，补齐本轮模型调用需要的上下文：model、provider、messages、tools、callbacks。
- （可忽略）从数据库恢复 User、Agent、Model、Provider、Tool 权限等上下文。
- 根据 provider 创建对应的 model runtime/client。
- provider client 由通用运行时工厂生成，支持两个关键扩展点：
  - `handlePayload`：请求前把内部统一 payload 转成厂商 API 参数。
  - `handleStream`：响应后把厂商 stream chunk 转成内部统一流协议。
- provider 返回的流会被转成 Web 标准 `ReadableStream`。
- 流经过多个 `TransformStream`：
  - 归一化 chunk 类型：text、reasoning、tool_calls、usage、stop。
  - 触发 callbacks：onText、onThinking、onToolCall。
  - 输出给上层消费或 SSE。
- `call_llm` 必须消费整个 Response body，驱动流持续读取。
- callbacks 中累积文本、reasoning、tool_calls，并发布 `stream_chunk`。
- 流结束后发布 `stream_end`，返回 `llm_result` 给 Runtime/Brain，决定下一步是 finish 还是 call_tool。


## 将 Chunk 流包装成 Stream 对象 📌

适配器的话在上一章讲，目前我们来讨论如何规范化输出 SSE 流。

正常来说，OpenAI 等 SDK 返回给用户或者开发者的只是被包裹在 `response.body` 里的一串数据。它会在每次我们调用 `read` 方法时返回数据；如果当前客户端协议层中的缓存满了，则会暂停和大模型的通信，直到开发者继续调用 `read` 来对缓存池进行泄压。

可以看出来，`response` 是一种单向的、不可回溯的数据。这种数据如果需要调用肯定非常麻烦，所以这里需要我们来进行一次包装。

目标：

- 合理的 Stream 流。

1. 一个合理的 Stream 流首先需要针对的是一个迭代器，迭代器提供 `next` 方法，可以让 Stream 流进行可控的拉取。所以我们要保证进来的是一个迭代器。
2. 保证接收到的数据流是一个可迭代器以后，接下来我们就可以简单地对其进行包装，包装成一个 `ReadableStream` 对象。
3. 手动添加 `ReadableStream` 对象的 `pull`、`cancel`、`start` 方法。
4. 保证 `pull` 方法触发了迭代器的 `next` 方法。
5. 使用 `ReadableStream` 对象自带的 `controller` 来进行关闭和入列。
6. 当触发到迭代器触底时，执行 `close`。

```typescript
export const convertIterableToStream = <T>(
  stream: AsyncIterable<T>,
) => {
  const iterable = chatStreamable(stream); // 冗余代码
  const it = iterable[Symbol.asyncIterator](); // 转换为迭代器

  return new ReadableStream<T>({
    async cancel(reason) {
      await it.return?.(reason); // 关闭，reason 是消费者传入的
    },
    async pull(controller) {
      try {
        const { done, value } = await it.next(); // 取出迭代器返回的对象
        if (done) controller.close(); // 判断迭代器是否结束
        else controller.enqueue(value); // 入队，实际上消费者调用 read 只是拿到队列里最新的数据，每次 pull 都是根据队列里还有没有数据来进行触发的
      } catch (e) {
        controller.enqueue(e);
        controller.close();
      }
    },
    async start(controller) {
      try {
        const { done, value } = await it.next(); // start 其实也是做一样的事
        if (done) controller.close();
        else controller.enqueue(value);
      } catch (e) {
        controller.enqueue(e);
        controller.close();
      }
    },
  });
};
```

以上就是一个很简洁的 SSE 流包装。

`const it = iterable[Symbol.asyncIterator]();` 这一行是专门获取异步迭代器的固定写法。


## 包装成合理的 Streamable 消费对象 🧩

仅仅是让消费者可以合理地拿到数据还不够，我们还需要针对数据流进行格式化。

由于上一步我们已经成功创建了一个可以使用的 `ReadableStream` 对象，我们在这一步可以使用所有 Stream 对象都可以调用的 `pipeThrough()` 方法。

该方法可以接受一个 `TransformStream` 对象，以一种链式调用的方式对接收到的流进行转换，或者执行一对一的读写任务。（注意：如果流被转换，则会导致锁定，其他对象将无法读取这个流。）

`pipeThrough(transformStream)`

`pipeThrough` 很简单，只是一个入口。接下来我们需要说一下 `TransformStream` 是如何运作的。

什么是 `TransformStream` 对象？

这是一个浏览器或者 Node.js 内置的对象，通常通过 `new TransformStream` 来进行创建。它一般只用于服务 `pipeThrough` 方法，提供对管道流的操作，可以针对流进行转换或者格式化。它暴露出以下属性进行调用：

- `start(controller)`：顾名思义，就是在流开始前进行触发。
- `transform(chunk, controller)`：对每一帧流进行转换操作，暴露出的 `chunk` 就是帧对象。
- `flush(controller)`：流结束时进行触发。

简单的实例代码：

```typescript
const textEncoderStream = new TransformStream({
  transform(chunk, controller) {
    console.log('[transform]', chunk);
    controller.enqueue(new TextEncoder().encode(chunk));
  },
  flush(controller) {
    console.log('[flush]');
    controller.terminate();
  },
});
```


# 规范 Stream 对象的 Transformer 🔄

OpenAI SDK 的输出通常不太适合直接阅读，格式类似下面这样：
```json
{
  "id": "chatcmpl-12345",
  "object": "chat.completion.chunk",
  "created": 1718888888,
  "model": "qwen-max",
  "choices": [
    {
      "index": 0,
      "delta": {
        "content": "好",
        "reasoning_content": "用户在问好，我应该回复你好..."
      },
      "finish_reason": null
    }
  ]
}
```
字段解释：

| 字段名                     | 中文含义 | 核心架构与原理解析                                                                                                                                                                                      |
| :---------------------- | :--- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`choices`**           | 选项数组 | **为什么是数组？** OpenAI API 设计允许在请求中传入 `n=3`，以一次性生成多个不同的回答选项。<br>**取值规范**：在绝大多数业务和流式对话场景下，`n` 默认为 1，我们只需要一个回答，因此代码中永远固定读取 `choices[0]`。                                                             |
| **`delta`**             | 数据增量 | **物理意义**：在数学和物理中代表“增量”或“变化量”。<br>**为什么不用 message？** 非流式接口一次性返回完整文本，字段叫 `message`。而在流式（SSE）传输下，为了节省网络带宽，模型每几十毫秒仅下发新生成的几个字，绝不会把之前的全量文本重传。这个**新产生的文本碎片**，就叫 `delta`。                              |
| **`content`**           | 正文内容 | 属于 `delta` 内部属性。存放模型输出给用户的最终文字碎片（如一个“好”字）。前端通过将成百上千个 chunk 里的 `content` 持续拼接，最终形成了完整的打字机动画效果。                                                                                                  |
| **`reasoning_content`** | 思考增量 | 属于 `delta` 内部属性（**非 OpenAI 标准，属于扩展定义**）。DeepSeek/Qwen 等推理模型专有，用于存放得出最终答案前的一段内部思考过程。这部分文字对用户不可见（或折叠在 UI 里），在数据结构上与 `content` 完全平级。                                                              |
| **`finish_reason`**     | 结束原因 | **判断请求生命周期是否终结的唯一安全标准**：<br>• **持续输出时**：值永远为 `null`。<br>• **`"stop"`**：模型正常且完整地回答完毕。<br>• **`"tool_calls"`**：模型认为现在必须调用工具，主动停止文本生成，把控制权交还给业务代码。<br>• **`"length"`**：代表触发了 Token 最大长度限制，回答被迫截断。 |
我们需要将其翻译成类似 `{ id: xx, type: 'xx', data: 'xx' }` 的结构。这种结构更便于阅读，因此需要一个专门的转换函数：
```ts
export const transformMinimalStream = (chunk: any): ChatStreamProtocolChunk | ChatStreamProtocolChunk[] => {
    // 1. 拦截 Token 使用量 (如果有计费对账需求，直接透传原生的 usage)
    if (chunk.usage && chunk.choices?.length === 0) {
        return { type: 'usage', data: chunk.usage };
    }

    const item = chunk.choices?.[0];
    if (!item) {
        return { type: 'data', data: chunk, id: chunk.id };
    }

    const delta = item.delta;

    // 2. 核心：处理工具调用 (Tool Calls)
    // 注意：面试中如果问到流式 Tool Call，你要知道此时 delta.tool_calls 是一个碎片数组
    // 目前直接原样透传，等后面我们在 Executor 链路中处理工具执行时，再去拼接这些碎片
    if (delta?.tool_calls) {
        return { type: 'tool_calls', data: delta.tool_calls, id: chunk.id };
    }

    // 3. 核心：处理 DeepSeek/Qwen 的思考流
    if (delta && 'reasoning_content' in delta && delta.reasoning_content) {
        return { type: 'reasoning', data: delta.reasoning_content, id: chunk.id };
    }

    // 4. 核心：处理常规打字机文本
    if (typeof delta?.content === 'string') {
        return { type: 'text', data: delta.content, id: chunk.id };
    }

    // 5. 核心：处理流结束信号 (非常重要，用于下游判断请求完毕)
    if (item.finish_reason) {
        return { type: 'stop', data: item.finish_reason, id: chunk.id };
    }

    // 兜底：未知格式原样返回
    return { type: 'data', data: chunk, id: chunk.id };
};
```
这个函数并不复杂，主要负责判断不同场景，可以在实际接入时按需开发。

## 将 Chunk 流格式化为 SSE 可用的纯文本 📡

接下来我们就需要使用这个管道 `transformer` 方法来针对刚才封装好的流进行转换。现在大模型返回的还是 `chunk` 对象，而不是 SSE 协议可以消费的纯文本（SSE 只能接受纯文本传输）。
**所以我们需要将传输进来的对象转换成纯文本：**

```typescript
export const streamSSETransformer = (transform?: (chunk: any) => ChatStreamProtocolChunk | ChatStreamProtocolChunk[]) => {
    let hasTerminalEvent = false
    return new TransformStream({
        async flush(controller) {
            if (!hasTerminalEvent) {
                const id = 'stream_end';
                const data = {
                    body: { name: 'Stream parsing error', reason: 'unexpected_end' },
                    message: 'Stream ended unexpectedly',
                    name: 'Stream parsing error',
                    type: 'StreamChunkError',
                };
                controller.enqueue(`id: ${id}\n`);
                controller.enqueue(`event: error\n`);
                controller.enqueue(`data: ${JSON.stringify(data)}\n\n`);
            }
        },

        async transform(chunk, controller) {
            let result = transform ? transform(chunk) : transformMinimalStream(chunk) //这里直接调用我们刚才规范化输出的函数
            let buffer = Array.isArray(result) ? result : [result]
            buffer.forEach(({ id, type, data }) => {
                controller.enqueue(`id: ${id}\n`)
                controller.enqueue(`event:${type}\n`)
                controller.enqueue(`data:${JSON.stringify(data)}\n\n`)
                if (type === "stop") {
                    hasTerminalEvent = true
                }
            })
        }
    })
}

```

我们在 `transform` 方法里进行拍平。`transform` 前面讲过，是 Stream 流特有的方法，可以由管道方法注入，管道的每一帧都会被执行我们定义好的 `transform` 方法。

仔细看一下我们做了什么：实际上就是将传入的值拍成三个字符串chunk塞入队列。记得需要先把chunk流转换成我们需要的`{id:xx,data:xx,type:xx}`的格式

# Stream 对象的消费回调

不仅可以简单地拍平对象做到纯文本，我们还可以让方法执行我们想要的回调，这样可以在监听到需要的事件时执行对应的回调。

```typescript
export const streamCallbackTransformer = (callbacks?: ChatStreamCallback) => {
    const textEncoder = new TextEncoder();
    let realText = ''
    let realReasonText = ''
    let errorReason = ''
    let finishReason = ''
    let currentType = '' as unknown as ChatStreamProtocolChunk['type']
    return new TransformStream({
        async flush() {
            const data = {
                text: realText,
                reasonText: realReasonText,
                errorReason,
                finishReason
            }
            if (callbacks?.onCompletion) {
                await callbacks.onCompletion(data)
            }
        },
        async start(controller) {
            if (callbacks?.onStart) {
                await callbacks.onStart()
            }
        },
        async transform(chunk: string, controller) {
            controller.enqueue(textEncoder.encode(chunk))

            if (chunk.startsWith('event:')) {
                currentType = chunk.split('event:')[1].trim() as ChatStreamProtocolChunk['type']
            }
            else if (chunk.startsWith('data:')) {
                let content = chunk.split('data:')[1].trim()
                const data = safeParseJSON(content)
                if (data === undefined) return

                switch (currentType) {
                    case 'text': {
                        realText += data
                        await callbacks?.onText?.(data as unknown as string)
                        break;
                    }

                    case 'reasoning': {
                        realReasonText += data
                        await callbacks?.onThinking?.(data as unknown as string)
                        break;
                    }
                    case 'stop': {
                        // handle stop event if needed
                        if (typeof data === 'string' && data && !finishReason) {
                            finishReason = data
                        }
                        break;
                    }
                    case 'error': {
                        errorReason = data as unknown as string;
                        await callbacks?.onError?.(errorReason)

                        break;
                    }
                    default:
                        break;
                }
            }
        }
    })
}

```
我们需要定义三个方法。注意，这是最重要的流处理函数之一：
1. 它会负责将我们在执行器里定义好的回调函数与流绑定在一起，以确保流在传输的时候我们的回调会被正确触发。
2. 我们还需要在结束的时候收集数据，触发对应回调。
3. 收集错误信息，确保这里的错误能被外部接收。

需要注意的是，这段代码由于是跟在刚才那段“将对象拍成文本”的转换器之后，所以我们无法通过对象的属性来判断当前的 `chunk` 是什么类型。

所以需要创建一个跨 `chunk` 的变量，利用闭包将它保存起来。这样它可以跨 `chunk` 读取；而且由于 SSE 文本格式的特性，一定会先出现 `event` 这个文本。
