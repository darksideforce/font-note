# 建立数据库

一个 Agent 最起码需要这几张表：`user`、`topic`、`message`、`operations`、`operations_events`。

- `user`：用户表，和对话进行关联。
- `topic`：对话表，存储对话的基础元信息。
- `message`：消息表，存储单次消息的所有信息。
- `operations`：后端状态机表，用于判断流转状态。
- `operations_events`：流式数据表，用于存储 Agent 产生的数据，用于回放。

# Drizzle 的数据库管理

前面讲过，Drizzle 使用的是一种 schema 的方式来进行管理。ORM 通用的做法就是通过自带语法编写一个数据库表声明文件。声明好对应的 schema 后，在 Drizzle 的客户端创建内进行引用即可。

```typescript
export const db = drizzle(postgresClient, { schema });
```

Drizzle 的 CLI 会去读取这段代码，并解析出所有用到的 schema，再根据这些配置项生成复杂的数据库建表以及改字段的 SQL 脚本。实际上，本质还是 Drizzle 来操作 SQL 脚本。

Drizzle 通过调用 `pgTable()` 来生成表，它接受表名以及一个配置项。配置项以 `key: {}` 的方式来生成表键和表值。

这里简单说一下一些函数的作用：

- `text()`：这里面是表的列名，比如 `userId: text('user_id')`，外面的键代表我们未来调用的时候传入的名。
- `primaryKey()`：代表这是主键。
- `notNull()`：代表这是必填。
- `references()`：代表这张表关联了哪张表。
- `timestamp()`：代表生成时间戳。
- `jsonb()`：代表这里可以接受 JSON 二进制数据。

# 基础的 Agent 需要什么数据模型

我先说明一下层级关系：

用户 => 对话 => 消息 => 状态机 => 推流事件，这就是最小的 Agent 需要的数据模型，多了的我们暂时不讨论。

- 用户是最基本的结构，Agent 不可能只有一个用户。用户负责维护 `id`、账户名、密码、最后一次活跃时间等和用户有关的信息，这一块我们完全可以依赖于 Better Auth 建立的用户表。
- `topic` 对话表，该表负责维护每个对话相关的 `id`、`user_id`、`title`、`status`、`agent_id`、`summary`、`metadata`、`created_at`、`updated_at`、`archived_at` 等信息。
- 单个消息表，每个消息表通过 `topicId` 与 `topic` 表关联，维护 `id`、`conversation_id`、`user_id`、`role`、`content`、`parent_id`、`operation_id`、`model`、`provider`、`tools`、`error`、`metadata`、`created_at`。
- 状态机表，这个表和消息是属于从属关系，维护 `id`、`user_id`、`conversation_id`、`status`、`trigger`、`model`、`provider`、`started_at`、`completed_at`、`error`、`usage`、`cost`、`trace_key`、`metadata`。
- `operations_events`：前端最关心的表，维护 `id`、`operation_id`、`seq`、`type`、`payload`、`created_at`，每个事件只会关联到一次状态机。

你可能发现了，为什么对话表内并没有属性表示对话表内有多少条消息。原因是这样的，当数据库的数据并没有大到一定程度时，关系型数据库本身就可以将两张表通过 SQL 语句查询出来。我们完全可以通过使用 Drizzle 提供的 API 来查询 `message` 表中有多少条数据的 `topicId` 是我们需要查询的值。
