# 为什么这样做
有时候我们需要每一次的store都是独立的，而每个页面之间的store不会互相干扰。比如每一次的对话都需要用到独属于自己的一系列store。如果数据很多很繁杂的时候我们也需要使用这种架构来进行维护

# 使用架构分离
- 使用实例工厂来进行创建一个工厂方法
- 使用一个工具函数来进行解构出provider和hook钥匙
- 在真实的组件中，使用一个provider来组装工厂和provider
- 使用selectors来减少数据维护压力
 **创建工厂方法**：
 ```ts
 export const createStoreAction: CreateStore =
  ({ context, hooks = {}, skipFetch }) =>
  (...params) => ({
    ...initialState,
    context,
    hooks,
    skipFetch,
    // ===== Slices =====
    ...dataSlice(...params),
  }); 
	//外部的
	export const createStore = (params: CreateStoreParams) =>
  createWithEqualityFn(devtools(createStoreAction(params)), shallow);	
	
 ```
我们可以把createstore直接看成一个经过了函数柯里化的工厂函数，这个函数可以视为连续接受两种参数，第一种参数可以视为我们在provider内调用时传入的参数，第二种参数则是创建zustand是传入的get和set

 在非react组件中使用selectors：
 react和vue一个很大的地方也是让我很意外的地方就是，像zustand的hooks，其实只被允许在react中进行使用，如果我们需要在其他地方，像class或者其他纯函数的地方来进行使用，就需要使用到zustand提供出来的一种钩子：getState，这种钩子将会返回调用时的store快照
 ```ts
 export const getAgentStoreState = () => useAgentStore.getState();
 ```
 一般来说，我们需要封装一下该api。
 使用也可以直接进行使用
 ```ts
 getAgentStoreState().agentList[agentId]
 ```
 也可以通过selectors的方式来进行调用
 注意这里又是一个柯里化的方式，我们都知道selectors其实通常需要接受一个store对象来进行操作，这样的操作就是取出一个selectors，并传入我们需要的id，该函数返回一个真正的selectors以接受store
 ```ts
 agentSelectors.getAgentConfigById(agentId)(getAgentStoreState())
 //函数部分
 const getAgentConfigById =
  (agentId: string) =>
  (s: AgentStoreState): LobeAgentConfig =>
    s.agentMap[agentId] as LobeAgentConfig;
```
